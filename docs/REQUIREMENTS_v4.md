## 1. 프로젝트 정의 및 비전

### 1.1. 한 줄 정의

OpenRAG-Lab은 **사용자가 자신의 PC에서 개인 문서로 RAG 시스템을 만들고, 그 품질을 정량적으로 비교·튜닝할 수 있는 로컬 워크벤치**다.

### 1.2. 핵심 가치 제안 (Value Proposition)

네 가지 가치를 동시에 제공한다.

1. **프라이버시**: 문서, 임베딩, 질의·응답이 외부 서버로 나가지 않는다. 외부 LLM API는 사용자가 명시적으로 활성화한 경우에만 호출한다.
2. **투명성**: 청킹 결과, 검색된 컨텍스트, 평가 점수 등 RAG 파이프라인 내부의 의사결정 과정을 모두 가시화한다.
3. **반복 실험성**: 동일 데이터셋에 대해 여러 설정 조합(모델, 청크 크기, 검색 알고리즘)을 한 번에 돌리고 결과를 비교할 수 있다.
4. **세 OS에서 동일하게 동작**: macOS, Windows, Linux를 동등 지원(1급 시민)한다. 모든 P0 기능이 세 OS에서 동일한 사용자 경험을 제공한다. 상세 정책은 `PLATFORM.md` 참조.

### 1.3. 본 프로젝트가 하지 않는 것 (Non-Goals)

명시적으로 범위에서 제외하여 스코프 크리프를 방지한다.

- 클라우드 기반 멀티유저 협업 환경 구축
- 사내 SSO·권한 관리 시스템
- 대규모 LLM의 자체 파인튜닝 기능
- 모바일 네이티브 앱
- 실시간 스트리밍 데이터(로그, 메시지) 인덱싱

## 2. 사용자 페르소나 및 사용 시나리오

각 페르소나는 추상적 페인포인트가 아닌 **구체적 사용 시나리오**로 정의한다.

### 2.1. 일반 사용자 — "지수" (The Pragmatist)

- **상황**: 변호사. 진행 중인 사건 관련 PDF·판례 자료 약 200건 보유.
- **목표**: "이번 분쟁 사례에서 임대차 갱신 거절 사유로 인정된 판례가 있나?"를 자연어로 검색하고, 근거 문장과 출처 페이지를 즉시 확인.
- **이상적 흐름**:
  1. 폴더를 통째로 드래그-앤-드롭한다.
  2. 'Auto-Pilot' 한 번 클릭.
  3. 5분 이내에 채팅 가능 상태가 된다.
  4. 답변에는 인용 문장과 출처 파일명·페이지 번호가 함께 보인다.
- **성공 기준**: 환경 설정·모델 선택·파라미터 조정 없이 채팅까지 도달.

### 2.2. RAG 학습자 — "민호" (The Learner)

- **상황**: 부트캠프 수강생. RAG 강의를 들었지만 청크 크기가 답변에 어떤 영향을 주는지 체감하지 못함.
- **목표**: 청크 크기를 256 → 512 → 1024로 바꿨을 때 같은 질문에 대한 답변이 어떻게 달라지는지 한눈에 비교.
- **이상적 흐름**:
  1. 학습용 샘플 데이터셋(예: 위키피디아 일부)을 선택한다.
  2. '청킹 비교' 화면에서 슬라이더로 크기를 조정한다.
  3. 동일 문서가 색상별로 어떻게 분할되는지 시각적으로 본다.
  4. 같은 질문을 세 가지 설정에 동시에 던지고, 답변과 검색된 컨텍스트를 나란히 본다.
- **성공 기준**: "청크 크기가 작을수록 정확도가 올라가지만 컨텍스트 부족 시 답변이 빈약해진다"를 시각적으로 이해.

### 2.3. 엔지니어 — "수진" (The Researcher)

- **상황**: 사내 챗봇 개발자. 자사 매뉴얼 1만 페이지에 가장 적합한 임베딩 모델·검색 전략 조합을 찾아야 함.
- **목표**: 임베딩 모델 3종 × 검색 전략 3종 = 9개 조합을 한 번에 평가하고, 정량 지표 표로 보고서에 첨부.
- **이상적 흐름**:
  1. 'Lab Mode'에서 후보 조합을 매트릭스로 정의한다.
  2. 평가용 골든 셋(질문 50개)을 업로드하거나 자동 생성한다.
  3. 일괄 실행 후 점수표·레이더 차트를 PNG/CSV로 내보낸다.
  4. 채택한 조합의 설정을 YAML로 익스포트하여 운영 환경에 반영한다.
- **성공 기준**: 9개 조합 벤치마킹을 클릭 몇 번으로 완료하고 결과를 그대로 보고서에 첨부 가능.

## 3. 기능 요구사항

각 기능은 **`기능명` → 목적 → 동작 정의 → 완료 기준(Acceptance Criteria) → 우선순위** 구조로 기술한다.

### 3.1. 환경 진단 및 자동 설정

#### 3.1.1. 하드웨어 프로파일링 `P0`
- **목적**: 사용자의 PC 사양을 자동 인식하여 실행 가능한 모델 범위를 결정.
- **동작**: 시스템 시작 시 다음 항목을 수집한다 — CPU 코어 수, 사용 가능 RAM, GPU 벤더 및 VRAM, OS 종류.
- **완료 기준**:
  - 시작 후 10초 이내에 진단 결과 카드가 화면에 표시된다.
  - GPU 미감지 시에도 CPU 모드로 정상 동작한다.

#### 3.1.2. 프리셋 추천 `P0`
- **목적**: 비전문가가 모델 선택의 부담 없이 즉시 시작할 수 있도록 함.
- **동작**: 진단 결과를 기준으로 다음 3가지 프리셋 중 가용한 것만 활성화 표시한다.
  - **속도 우선**: 경량 임베딩 모델, 작은 청크, 단순 검색.
  - **밸런스**: 중간 크기 모델, 표준 청크, 하이브리드 검색.
  - **정확도 우선**: 고성능 모델, 시맨틱 청킹, 리랭커 적용.
- **완료 기준**: 사용자가 카드 한 번 클릭으로 모든 파라미터가 자동 세팅된다.

#### 3.1.3. 의존성 체크 `P1`
- **목적**: 사용자가 누락된 라이브러리 때문에 막히지 않도록 안내.
- **동작**: 필요한 백엔드 라이브러리의 가용 여부를 체크하고, 누락 시 설치 가이드 링크를 제시한다.
- **완료 기준**: 사용자에게 "어떤 라이브러리가 왜 필요한지"가 명확한 메시지로 표시된다.

---

### 3.2. 데이터 관리 및 전처리

#### 3.2.1. 멀티 포맷 입력 `P0`
- **목적**: 사용자가 포맷 변환 없이 보유 문서를 그대로 투입할 수 있도록 함.
- **동작**: 다음 포맷을 표준 텍스트(+메타데이터)로 변환한다.
  - **MVP 필수**: PDF, TXT, Markdown.
  - **MVP 이후**: DOCX, HTML, JSON, EPUB.
- **완료 기준**:
  - 파일 단위·폴더 단위 드래그-앤-드롭이 모두 동작한다.
  - 변환 실패 파일은 별도 리스트로 표시되며 전체 작업이 중단되지 않는다.

#### 3.2.2. 문서 구조 보존 파싱 `P1`
- **목적**: 표·이미지 캡션 등 비텍스트 정보를 검색 대상에 포함.
- **동작**:
  - 표는 마크다운 표 형식으로 직렬화하여 청크에 포함한다.
  - 이미지의 alt 텍스트·캡션이 있는 경우 별도 청크로 보존한다.
  - 페이지 번호·섹션 제목을 메타데이터로 함께 저장한다.
- **완료 기준**: 검색 결과에 "OO.pdf, p.42, 표 3.1" 같은 출처가 정확히 표시된다.

#### 3.2.3. 청킹 실험실 `P0`
- **목적**: 사용자가 청크 분할 전략이 검색 품질에 미치는 영향을 직접 체감할 수 있도록 함.
- **동작**: 다음 4가지 전략을 선택할 수 있다.
  - 고정 크기 (Fixed) — `P0`
  - 재귀적 분할 (Recursive) — `P0`
  - 문장 단위 (Sentence) — `P1`
  - 시맨틱 (Semantic) — `P1`
- **완료 기준 (P0)**:
  - 청크 크기 슬라이더 조정 시 동일 문서가 색상별로 어떻게 나뉘는지 1초 이내에 미리보기에 반영된다.
  - 청크 개수와 평균 토큰 수가 실시간으로 함께 표시된다.

### 3.3. 검색 및 응답 생성

#### 3.3.1. 임베딩 모델 매니저 `P0`
- **목적**: 비전문가도 모델을 안전하게 선택·다운로드.
- **동작**:
  - 사전에 큐레이션된 모델 목록을 카드 형태로 보여준다 (이름·언어·크기·VRAM 요구치).
  - 다운로드 진행률을 표시하고, 한 번 받은 모델은 로컬 캐시에서 재사용한다.
- **완료 기준**: 사용자는 모델 이름이나 Hugging Face 경로를 직접 타이핑할 필요가 없다.

#### 3.3.2. 검색 알고리즘 `P0` / `P1`
- **목적**: 데이터 특성에 맞는 검색 전략 선택.
- **동작**:
  - **P0**: Dense Retrieval (벡터 유사도 검색).
  - **P1**: Sparse Retrieval (BM25), Hybrid Search (RRF).
- **완료 기준**: 동일 질문에 대해 검색 전략을 바꾸면 검색된 컨텍스트가 즉시 다르게 표시된다.

#### 3.3.3. 리랭커 `P2`
- **목적**: 검색 결과의 순위를 정교화.
- **동작**: 1차 검색 결과 상위 N개에 대해 리랭킹 모델을 적용하고, 적용 전후 순위 변화를 보여준다.
- **완료 기준**: 리랭커 ON/OFF 토글로 답변 변화를 즉시 비교할 수 있다.

#### 3.3.4. 검색 전용 모드 (Retrieval-Only Mode) `P0`
- **목적**: LLM 없이 임베딩·검색 품질만 평가하고 싶은 사용자를 지원. 임베딩 모델·청킹 전략을 비교할 때 LLM 변수를 제거하면 더 명확하게 검색 성능 자체를 볼 수 있다. 또한 저사양 PC나 LLM 미설치 환경에서도 워크벤치를 사용할 수 있게 한다.
- **동작**:
  - 워크스페이스 설정에서 `llm_id`가 비어 있으면 자동으로 검색 전용 모드 활성화.
  - 채팅 화면에서 답변 생성 단계는 생략되고, 검색된 청크 목록만 표시 (출처·점수·내용).
  - 평가 지표 중 LLM 의존 지표(Faithfulness, Answer Relevance)는 비활성화되고, 검색 단계 지표(Context Precision/Recall)만 산출.
- **완료 기준**:
  - 사용자가 LLM을 지정하지 않고도 인덱싱→검색→점수 산출 흐름을 끝까지 통과할 수 있다.
  - UI에 "검색 전용 모드" 배지가 표시되어 사용자가 현재 상태를 인지할 수 있다.

#### 3.3.5. 외부 LLM 제공자 `P1`
- **목적**: 로컬 LLM의 성능 한계를 보완하기 위해 메이저 외부 LLM API를 선택적으로 사용. 단, 컨셉 §1.2 프라이버시 가치를 보호하기 위해 명시적 옵트인 절차를 거친다.
- **지원 제공자 (MVP 이후 P1)**: OpenRouter, Google Gemini, OpenAI, Anthropic.
- **동작**:
  1. 외부 LLM 사용 시도 시 시스템은 해당 제공자의 API 키가 등록되어 있는지 우선 확인한다.
  2. 키가 없으면 사용자에게 등록 화면을 보여주며, 코드는 키가 등록될 때까지 대기 (자동 진행 금지).
  3. 키 등록 후 사용자가 명시적으로 "이 워크스페이스에서 OOO 제공자 사용을 허용" 토글을 켜야 호출 가능.
  4. 외부 호출 발생 시 UI 상태 표시줄에 명시 (어떤 제공자, 어느 단계).
- **완료 기준**:
  - 키 미등록 시 자동으로 호출하지 않고 등록 화면을 띄운다.
  - 워크스페이스별로 외부 사용 허용 여부를 독립적으로 설정 가능.
  - 모든 외부 호출이 UI에 가시화된다.

### 3.4. 평가 및 벤치마킹

#### 3.4.1. 골든 셋 관리 `P0` / `P1`
- **목적**: 답변 품질을 정량적으로 측정할 기준 데이터 확보.
- **동작**:
  - **P0**: 사용자가 질문-기대답변 쌍을 직접 입력하거나 CSV로 업로드.
  - **P1**: 업로드된 문서를 기반으로 LLM이 자동 생성 (사용자 검수 단계 포함).
- **완료 기준**: 최소 10개 이상의 평가 쌍을 등록·편집·삭제할 수 있다.

#### 3.4.2. 정량 지표 측정 `P0`
- **목적**: 주관적 인상이 아닌 숫자로 품질을 비교.
- **동작**: 다음 지표를 자동 계산한다.
  - **Faithfulness**: 답변이 검색된 컨텍스트에 근거하는가.
  - **Answer Relevance**: 답변이 질문에 대응하는가.
  - **Context Precision/Recall**: 검색 단계가 필요한 정보를 잘 가져왔는가.
- **완료 기준**: 평가 실행 후 4개 지표가 0~1 사이 점수로 표시되고, 항목별 산출 근거(어떤 문장이 어떻게 평가됐는지)를 클릭하여 확인할 수 있다.

#### 3.4.3. A/B 비교 대시보드 `P0`
- **목적**: 두 설정 중 어느 쪽이 더 나은지 한눈에 결정.
- **동작**: 좌·우 두 설정의 지표를 동일 차트에 겹쳐 표시한다.
- **완료 기준**: "설정 A가 Faithfulness에서 +0.12 우세, Latency에서 -0.3초 우세" 같은 요약 라벨이 자동으로 붙는다.

#### 3.4.4. 성능 프로파일러 `P1`
- **목적**: 정확도뿐 아니라 속도·자원 소모도 평가.
- **동작**: 단계별 소요 시간(파싱/임베딩/검색/생성)과 메모리·VRAM 사용량을 기록.
- **완료 기준**: 각 단계의 시간 비중이 막대 그래프로 표시된다.

---

### 3.5. 서비스 운영

#### 3.5.1. 내장 채팅 인터페이스 `P0`
- **목적**: 별도 도구 없이 즉시 사용 가능한 환경 제공.
- **동작**: 좌측 채팅, 우측 검색된 컨텍스트(인용 출처 포함) 패널이 함께 표시된다.
- **완료 기준**: 답변의 모든 주장에 대해 어떤 청크에서 왔는지 클릭으로 추적 가능하다.

#### 3.5.2. 설정 익스포트 `P0`
- **목적**: 튜닝 결과의 재현성과 이식성 확보.
- **동작**: 현재 활성화된 모든 파라미터를 단일 YAML 파일로 저장.
- **완료 기준**: 같은 YAML을 다른 PC에서 불러와도 동일한 파이프라인이 재구성된다.

#### 3.5.3. 로컬 API 서버 `P1`
- **목적**: 외부 애플리케이션에서 사용.
- **동작**: 튜닝된 설정 그대로 RESTful 엔드포인트로 노출 (Swagger 포함).
- **완료 기준**: `curl` 한 줄로 검색 + 답변을 받을 수 있다.

#### 3.5.4. Dockerfile 생성 `P2`
- **목적**: 동일 환경을 다른 서버에 배포.
- **동작**: 현재 워크스페이스를 그대로 컨테이너화할 수 있는 Dockerfile을 생성.
- **완료 기준**: 생성된 Dockerfile로 빌드한 컨테이너가 로컬과 동일하게 동작한다.

## 4. 비기능 요구사항

추상적 표현 대신 측정 가능한 기준으로 정의한다.

### 4.1. 보안 및 프라이버시
- 외부 네트워크 호출은 다음 두 가지 경우 외에 발생해서는 안 된다.
  1. 사용자가 명시적으로 활성화한 외부 LLM API 호출.
  2. 사용자가 명시적으로 요청한 모델 다운로드.
- 외부 호출이 발생하는 모든 시점은 UI 상태 표시줄에 명시한다.

### 4.2. 성능 (참고 기준 — 검증 후 확정)
- 100MB 분량의 PDF 인덱싱이 미들레인지 GPU(VRAM 8GB) 환경에서 10분 이내 완료.
- 검색 + 답변 생성의 총 응답 시간이 10초 이내.
- 위 수치는 MVP 기간 중 실측을 통해 확정한다.

### 4.3. 사용성
- 첫 실행부터 첫 채팅 답변까지 비전문가가 5분 이내에 도달 가능.
- 모든 파라미터에 마우스 오버 시 1~2문장 설명이 표시된다.
- CLI 로그와 GUI가 동기화되어 진행 상황을 양쪽 어디에서든 확인 가능.

### 4.4. 확장성
- 임베딩 모델·벡터 DB·청킹 전략은 어댑터 인터페이스로 분리되어, 새 항목 추가 시 기존 코드를 수정하지 않아도 된다 (구체 인터페이스는 기술 스택 문서에서 정의).

### 4.5. 복구성
- 인덱싱 중단 후 재실행 시, 이미 처리된 문서는 건너뛰고 미처리 분부터 재개한다.
- 임베딩 캐시는 (문서 해시 + 모델 ID + 청킹 설정)을 키로 관리한다.

## 5. 사용자 흐름

### 5.1. 비전문가 흐름 (Auto-Pilot)
```

```
중간에 어떤 선택지도 강제로 띄우지 않는다.

### 5.2. 학습자 흐름 (Comparison)
```
실행 → 샘플 데이터셋 선택 → 청킹 슬라이더 조정 (실시간 미리보기)
     → 동일 질문을 여러 설정에 동시 질의 → 답변·컨텍스트 나란히 비교
```

### 5.3. 엔지니어 흐름 (Lab Mode)

```
     → 일괄 평가 실행 → 스코어카드 확인
     → 최종 조합 선택 → 전체 데이터 인덱싱 → YAML/API 익스포트
```

## 6. 데이터 및 상태 관리

### 6.1. 워크스페이스 모델
- **워크스페이스(Workspace)**: 하나의 프로젝트 단위. 다른 워크스페이스의 데이터·설정과 완전히 격리된다.
- 워크스페이스에는 다음이 포함된다.
  - 원본 문서 참조 (실제 파일은 외부 경로로 링크 가능)
  - 청킹·임베딩 캐시
  - 벡터 인덱스
  - 실험 이력 (설정 + 평가 결과)
  - 골든 셋

### 6.2. 실험 이력 추적
- 모든 실험은 (워크스페이스 ID, 설정 스냅샷, 평가 점수, 실행 시각)을 한 단위로 기록한다.
- 사용자는 과거 실험을 언제든 불러와 재현하거나 비교할 수 있다.

### 6.3. 캐시 정책
- 임베딩 캐시는 키 기반(문서 해시 + 모델 ID + 청킹 파라미터)으로 관리되어, 설정이 동일하면 재계산을 생략한다.
- 캐시 크기 한도와 청소 정책은 사용자 설정으로 노출한다.

## 7. MVP 범위 요약 (2-4주)

다음만 MVP 범위에 포함한다.

| 영역 | 포함 (P0) | 제외 (P1·P2) |
|---|---|---|
| 환경 진단 | 하드웨어 프로파일링, 프리셋 추천 | 의존성 자동 설치 |
| 입력 | PDF·TXT·Markdown | DOCX·HTML·JSON·EPUB, 표/이미지 보존 |
| 청킹 | 고정 크기, 재귀적 분할 | 시맨틱, 문장 단위 |
| 검색 | Dense (벡터 검색), 검색 전용 모드 | BM25, Hybrid, 리랭커 |
| LLM | 로컬 LLM (선택), LLM 없이도 동작 | 외부 LLM 4종 (OpenRouter/Gemini/OpenAI/Anthropic) |
| 평가 | 수동 골든 셋, Context Precision/Recall (검색 전용 모드 포함), Faithfulness·Answer Relevance (LLM 있을 때), A/B 비교 | 자동 골든 셋 생성, 프로파일러 |
| 운영 | 채팅 UI, YAML 익스포트 | API 서버, Dockerfile |

이 범위가 끝나면 다음 세 가지 핵심 시나리오가 모두 동작한다:
1. "비전문가가 PDF를 넣고 채팅하기" (로컬 LLM 사용)
2. "엔지니어가 두 설정을 정량 비교하기" (Faithfulness·Relevance 포함)
3. "LLM 없이 임베딩·검색 성능만 평가하기" (검색 전용 모드)

## 10. 추가 결정 사항 (설계 단계에서 확정)

설계서에서 도출된 결정사항도 함께 기록한다.

5. **임베딩 차원 변경 시 재인덱싱 정책** → **결정됨 (v4)**:
   - **3단계 처리**:
     1. **사전 감지**: YAML 임포트나 임베더 변경 시 차원 비교. 다르면 즉시 경고 (`EMBEDDER_DIM_MISMATCH`).
     2. **명시적 동의**: 모달로 "기존 인덱스가 무효화되며 전체 재인덱싱이 필요합니다 (예상 시간 X분). 계속하시겠습니까?" → 사용자가 클릭해야 진행.
     3. **이전 실험 보존**: 새 임베더로 재인덱싱하더라도 **기존 실험 결과는 삭제하지 않고 archived 상태로 보존**한다. 사용자는 두 임베더의 결과를 비교 가능. 벡터 자체는 ChromaDB 컬렉션을 차원별로 분리하여 격리한다.
   - **이유**: 엔지니어 페르소나(수진)의 매트릭스 평가 핵심 시나리오가 "여러 임베더 비교"이므로, 차원이 다르다고 이전 결과를 폐기하면 비교가 불가능해진다.
   - 상세 명세: 설계서 §6.1 (워크스페이스 모델), API 명세서 §11 (실험 archived 상태).

6. **LLM 없는 환경의 동작** → **결정됨 (v4)**:
   - **검색 전용 모드 (Retrieval-Only Mode) 1급 지원**.
   - `llm_id`가 비어 있으면 자동 활성화. 채팅 화면은 검색된 청크 목록만 표시, 평가는 검색 단계 지표(Context Precision/Recall)만 산출.
   - 임베딩·검색 성능을 LLM 변수 없이 평가하고 싶은 사용자(특히 엔지니어 페르소나의 임베딩 모델 선택 단계)를 명시적으로 지원.
   - 상세 명세: §3.3.4, API 명세서 §9 (`answer: null` 허용), CONFIG_SCHEMA §4.3 (`llm_id: null` 의미).
